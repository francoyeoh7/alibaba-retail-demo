# 新店申请转老店升级设计说明

## 目标

为商家端“申请记录”补充一条从 `新店申请` 平滑切换到 `老店升级` 的补救链路，减少重复申请和资料二次填写，同时保证原单据关闭、点位释放、异常校验可解释。

## 场景假设

- 设计载体为 PC 端商家管理系统，基于 Ant Design 5.x。
- 用户需要在列表页快速识别哪些记录可转化，并完成一次低风险、可预期的转换动作。
- 本设计优先贴近现有企业后台风格，不做品牌级重设计。

## 方案对比

### 方案 A：列表页触发 + 确认弹窗 + 独立表单页

优点：
- 最贴近现有 PRD 和截图参考，评审理解成本低。
- 将“高风险确认”和“复杂补充表单”拆开，认知负担更低。
- 更适合在笔试中展示完整的正常流、异常流和提交结果。

缺点：
- 页面切换较多，需要在转场时明确字段来源与豁免规则。

### 方案 B：列表页触发 + 抽屉式补表单

优点：
- 用户停留在列表上下文，切换成本低。

缺点：
- 老店升级表单字段量大，抽屉承载拥挤，不利于展示信息来源和校验逻辑。
- 与当前截图中的已有申请页结构不一致，落地可信度较弱。

### 方案 C：列表页批量操作区合并“查看/转化”

优点：
- 可统一操作入口。

缺点：
- 需求仅针对极少量满足条件的单据，批量入口会抬高理解成本。
- 无助于强调不可逆风险，不适合当前任务。

## 推荐方案

采用方案 A：`申请记录列表 -> 转换确认弹窗 -> 老店升级申请表单 -> 提交成功反馈`。

## 页面与交互

### 1. 申请记录列表页

- 使用 `Table + Form + Tag + Button + Tooltip` 组织列表与筛选。
- 仅当 `审批状态=店铺信息待完善` 且 `申请类型=新店申请` 时展示主操作 `转老店升级`。
- 不满足条件的记录不展示该按钮，避免误导。
- 行内补充 `可转化` 标识，提升可发现性。

### 2. 转换确认弹窗

- 使用 `Modal + Radio/Select + Alert`。
- 字段：
  - `选择门店`：单选门店选择器，显示门店名、城市/区县、详细地址、法人。
  - `点位地址`：只读，来自原申请。
- 底部用红色风险提示说明“原始单据会自动关闭，不可逆”。
- 提交校验：
  - 门店地址为空：允许继续。
  - 门店地址与原申请地址一致：允许继续。
  - 地址不一致：阻断提交，页面级 `message.error` 提示具体地址差异。

### 3. 老店升级申请表单页

- 使用 `Card + Steps + Form + Upload + Descriptions + Alert + Affix`。
- 顶部展示转化来源摘要：
  - 原申请编号、申请类型、申请仓型、点位地址、租赁合同。
- 表单中明确标注字段来源：
  - `来自原新店申请`：申请仓型、点位地址、租赁合同。
  - `来自所选门店`：门店、城市、行政区、详细地址、法人信息、证照与公司信息。
- 页面中显示规则提示：本次转化申请跳过 `用户量` 与 `安全距离` 校验，其余校验照常执行。

### 4. 成功结果反馈

- 提交成功后显示 `Result` 或成功态消息。
- 同步更新列表数据：
  - 新增一条 `老店升级` 记录，状态为 `待平台审核`。
  - 原 `新店申请` 记录变更为 `已关闭` 且不可再次提交。
  - 在原记录操作列仅保留 `查看详情`。

## 异常与边界

1. 无可关联门店：弹窗内展示空状态，确认按钮置灰。
2. 门店地址不一致：阻断提交并给出具体地址提示。
3. 关联门店地址为空：允许继续，但在门店项中提示“地址缺失，将以原点位地址继续流程”。
4. 提交时缺少必填资料：表单字段级校验 + 顶部汇总提示。
5. 原记录已关闭或状态变化：返回列表后提示“该单据状态已变化，无法继续转换”。

## 设计难点与解决

### 难点 1：不可逆操作的确认成本

解决：
- 在列表页先限制按钮展示条件。
- 在弹窗中用只读点位地址和红色风险提示强化后果说明。
- 在确认前校验地址一致性，避免用户误把错误门店带入升级单。

### 难点 2：表单字段来源复杂，用户容易混淆

解决：
- 用分组卡片拆成“转化来源概览 / 点位信息 / 门店主体信息 / 审批规则提示”。
- 通过标签和辅助文案明确字段来源，减少“为什么这个字段不能改”的疑问。

### 难点 3：题目要求覆盖完整流程和异常态

解决：
- Demo 中内置多个 mock 场景：可转化成功、地址不一致、无门店可选、提交校验失败、提交成功后的列表回写。
- 用一个页面内的状态切换方式展示完整链路，方便笔试演示。
