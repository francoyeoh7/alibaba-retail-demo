# 设计说明文档

## 1. 设计目标

本次设计聚焦零售便利店平台里四条互相关联的效率优化/风险控制流程：

1. 商家原本提交了 `新店申请`，但在审批通过并进入 `店铺信息待完善` 后，发现已有符合资质的老店，希望直接转换成 `老店升级`，而不是重新发起一份新申请。
2. 服务商在做员工入离职维护时，容易误删/误停用承担协议签约与保证金缴纳职责的 `法人账号`，因此需要单独保护。
3. 校园场景订单在寒暑假、返校周、考试周会出现极端波动，直接按自然月统计会导致经营数据失真。
4. 运营人员在查看点位申请记录时，难以快速跳到对应审批详情，导致查询和处理链路割裂。

设计目标有六点：

1. 让用户在列表页快速识别“哪些单据能转、为什么能转”。
2. 让不可逆操作在确认前足够可预期，避免误关联门店。
3. 让转化后的老店升级表单具备明确的数据来源感，减少补填成本。
4. 让法人账号在员工维护场景中被清晰识别，并且无法被直接做危险操作。
5. 让校园订单统计能区分“真实经营变化”和“校历带来的季节性失真”。
6. 让运营后台从申请记录到审批详情形成一跳直达，而不是依赖人工检索审批单。

本次最终实现增加一条明确约束：

- 页面视觉和组件表达严格按照 `Ant Design 5.x 官方默认风格` 执行
- 不做品牌化渐变、玻璃感、大圆角强化、作品集式 Hero 区等二次设计
- 所有核心界面都回到官方中后台常见组合：`Layout + Breadcrumb + Card + Form + Table + Modal + Drawer + Result`

## 2. 页面与组件设计

### 2.1 申请记录列表页

使用组件：

- `Layout`：使用标准后台结构，不做额外视觉包装。
- `Table`：承载申请记录，保持 Ant Design 默认表格观感。
- `Button`：同时承载状态入口 `点位信息待完善`、主操作 `转老店升级` 和次操作 `查看详情`。
- `Tag`：标记审批状态，降低用户识别成本。
- `Form + Select + Input`：构成标准查询区，替代自定义筛选条。
- `Card`：承载查询区和表格区，符合官方列表页模式。

交互规则：

- 仅在 `审批状态=店铺信息待完善` 且 `申请类型=新店申请` 时显示 `转老店升级`。
- 满足条件的记录额外提供一个状态入口 `点位信息待完善`，用于承接参考图中的“先点状态再进入详情”的路径。
- 不满足条件的记录只展示 `查看详情`，避免出现灰置按钮带来的误导。
- Demo 场景切换被收敛到 `Collapse` 中，避免抢占正式页面视觉。

### 2.2 状态入口弹窗与详情页

使用组件：

- `Modal`：承接“点位信息待完善”的中间提醒层。
- `Descriptions`：展示待完善单据的关键摘要。
- `Steps`：在详情页中展示申请链路进度。
- `Alert`：说明当前处于待完善阶段，并提示可切换为老店升级。

交互规则：

- 用户点击状态入口 `点位信息待完善` 后，先看到中间弹窗，而不是直接跳表单。
- 中间弹窗的主按钮为 `查看申请详情`，进入 `新店申请详情` 页面。
- 详情页保留两条去向：
  - `继续完善新店申请`
  - `转老店升级`
- 这样既贴近题目中的两张参考图，也保留了原 PRD 的直接转换主动作。

### 2.3 转换确认弹窗

使用组件：

- `Modal`：承载高风险确认动作。
- `Radio`：单选关联门店，符合 PRD 中“单选框”的表达。
- `Descriptions`：展示原申请的只读点位地址。
- `Alert`：分别说明选择规则与不可逆风险。
- `message`：地址不一致时提供 toast 级错误反馈。

交互规则：

- 必须先选择一个门店，才能确认转换。
- 若门店地址为空，允许继续转换，并在后续表单中提示沿用原点位地址。
- 若门店地址与原申请地址不一致，阻断提交并 toast 提示：
  `所选门店地址“XX”与原审批通过地址不一致，无法转换`
- 确认成功后关闭弹窗，进入老店升级申请表单。

### 2.4 老店升级申请表单

使用组件：

- `Descriptions`：呈现转化来源概览。
- `Form`：承载补录信息和字段校验。
- `Input`：展示并编辑自动带出的门店主体信息。
- `Upload`：以附件形式展示原新店申请带出的租赁合同。
- `Alert`：说明本次转化跳过 `用户量` 和 `安全距离` 校验。

字段策略：

- `来自原新店申请`：申请仓型、点位地址、租赁合同。
- `来自关联门店`：门店、城市、行政区、详细地址、法人信息、统一社会信用代码、公司名称。
- 表单里将“点位字段”和“门店主体字段”分区展示，避免来源混淆。

校验规则：

- 地址一致性在弹窗阶段校验，不拖到表单阶段才报错。
- 法人手机号、公司名称、统一社会信用代码等主体信息仍遵循老店升级表单的必填逻辑。
- 对于 `门店资料缺失` 场景，允许用户在表单内补齐信息再提交。

### 2.5 提交成功反馈

使用组件：

- `Result`：承载成功态反馈。

状态变化：

- 生成一条新的 `老店升级` 记录，状态为 `待平台审核`。
- 原 `新店申请` 记录自动置为 `已关闭`。
- 原记录不再展示 `转老店升级` 操作。
- 成功文案中同步说明“点位释放”已处理，补足业务闭环。

### 2.6 员工维护与法人账号保护

使用组件：

- `Menu`：在左侧导航中纳入 `员工维护`，保证平台整体性。
- `Alert`：顶部持续提示当前法人账号是谁，以及为什么需要保护。
- `Table`：展示员工账号列表、在职状态、是否为法人账号。
- `Tag`：对 `法人账号` 做强标识。
- `Button + Popconfirm`：普通员工保留正常停用/删除动作，法人账号则禁用危险操作。
- `Modal + Form + Radio`：用于展示保护规则与完成法人账号移交。

交互规则：

- 法人账号行默认展示 `法人账号` 标签，且 `停用 / 删除` 按钮不可直接点击。
- 用户可通过 `查看保护规则` 理解影响范围，再通过 `立即发起移交` 进入移交流程。
- 在移交弹窗中选择新的接收人后，新法人账号立即接替签约与保证金权限；旧法人账号自动转为 `待停用`。

### 2.7 校园经营分析与假期影响校正

使用组件：

- `Menu`：将 `校园经营分析` 纳入统一平台导航。
- `Radio`：切换 `按日历月看 / 剔除假期影响后看`。
- `Button`：演示 `标准校正场景 / 校历缺失场景 / 官网导入后生效`，并提供 `校历管理` 入口。
- `Statistic`：突出展示原始订单量、参考订单量、可用于比较的营业天数、数据状态等核心指标。
- `Alert`：解释校正逻辑、回退逻辑和风险提示。
- `Descriptions + Table + Tag + Drawer + Collapse`：将校历来源、连续假期区间、官网导入结果和人工调整入口收进 `校历管理`。

交互规则：

- 默认进入 `按日历月看`，保留原始数据视角，同时提示当前看到的是原始订单。
- 切换到 `剔除假期影响后看` 后，若校历完整：
  - 输出参考订单量和可用于比较的营业天数
  - 主页面只展示经营结果和 `数据状态`，不直接堆放统计依据
  - 用户可通过 `校历管理` 查看学校来源、连续假期区间、校历版本和最近更新时间
- 若当前校区校历缺失：
  - 不输出伪造的校正值
  - 明确提示“学校假期信息还没配好，先按原始数据看”
  - 提供 `导入学校官网校历` 入口，导入成功后立即重算统计
- 校历区间按学校自己的连续日期识别，例如 `寒假 2026-01-20 至 2026-02-16`，不按自然月硬拆天数。
- 每一行明细都可以点击 `查看明细`，查看该校区当前月的日期拆分和处理方式。

### 2.8 审批管理与快捷跳转

使用组件：

- `Menu`：新增 `审批管理` 入口，体现平台存在运营后台角色。
- `Table`：承载点位申请记录与审批联动信息。
- `Tag`：标记当前申请阶段和同步状态。
- `Button`：提供 `查看审批详情` 快捷入口。
- `Drawer`：在不离开记录列表的前提下，直接展开审批详情。
- `Timeline + Descriptions + Alert`：展示审批流转记录、审批人、风险标签和当前状态。

交互规则：

- 每条已联动审批单的记录，都在 `审批单号` 和 `操作列` 提供快捷入口。
- 点击 `查看审批详情` 后，右侧 `Drawer` 直接展开，不需要跳转到独立系统。
- 若审批详情尚未同步：
  - 列表中显示 `审批详情待同步`
  - 操作按钮置为禁用态，避免点击无响应
- 这样既解决“找不到详情”的问题，也避免“有按钮但打不开”的假入口。

## 3. 关键状态覆盖

本次 Demo 覆盖了 12 类关键状态：

1. 正常转换成功。
2. 所选门店地址与原申请地址不一致，阻断转换。
3. 门店地址为空，允许继续并沿用原点位地址。
4. 当前账号无可关联门店，弹窗展示空状态并禁用确认。
5. 关联门店主体资料缺失，表单中补齐后再提交。
6. 从 `点位信息待完善` 状态入口进入中间弹窗，再进入详情页。
7. 员工维护页中法人账号被显式标记，危险操作默认禁用。
8. 法人账号移交成功后，新法人立即接替，旧法人账号进入 `待停用`。
9. 校园订单在自然月口径下展示原始波动。
10. 校园订单切到教学周校正口径后，恢复可比经营数据。
11. 校历缺失时，系统回退自然月统计并给出明确警告。
12. 运营后台申请记录可直达审批详情；待同步记录展示禁用态。

## 4. 设计难点与解决方案

### 难点一：不可逆动作如何做到“敢点但不乱点”

问题：
用户点击转换后，一旦成功，原单会关闭且不可逆，如果提示不清晰，容易造成心理负担或误操作。

解决：

- 先在列表页通过条件化按钮减少误触。
- 再在弹窗中展示原点位地址与不可逆风险。
- 把地址一致性校验提前到确认环节，让错误尽早暴露。

### 难点二：字段来源复杂，用户容易不知道哪些能改

问题：
同一表单中既有原申请带出的点位信息，也有关联门店带出的主体信息，如果没有层级划分，容易让用户误以为所有字段都要重新填写。

解决：

- 使用 `Descriptions + Form 分区` 明确来源。
- 为每组字段打上来源标签。
- 用辅助提示说明本次转化跳过的校验项，减少理解负担。

### 难点三：笔试要求覆盖异常态，但不能让 Demo 像“测试后台”

问题：
如果把所有异常都直接摊在一个页面里，会显得杂乱且不真实。

解决：

- 保留真实业务主界面不变。
- 把异常和边界状态折叠到右侧 `演示场景` 卡片中。
- 这样既方便演示，也不会破坏主流程的可信度。

### 难点四：员工维护问题不是这条 PRD 的主线，但又必须放在同一平台里解释清楚

问题：
用户指出“误删/误停用法人账号”也是本次平台迭代的一部分，如果单独拎成另一套页面，会割裂“零售便利店平台”作为整体系统的体验。

解决：

- 直接把 `员工维护` 放进同一套左侧导航。
- 保持与 `申请记录` 相同的 Ant Design 5.x 默认后台风格。
- 用统一 Layout 承载不同模块，让评审能看到这不是孤立原型，而是同一平台下的并列需求。

### 难点五：校园数据失真不是“报表样式问题”，而是统计口径问题

问题：
如果只做一张更好看的图表，并不能解决寒暑假停课导致的样本失真。真正的问题在于“该不该把寒暑假天数和教学期天数放在同一个分母里比较”。

解决：

- 把解决方案设计成“口径切换”，而不是单纯换图。
- 通过 `自然月口径 / 教学周校正口径` 明确告诉用户：系统提供的是两种统计视角。
- 当校历不完整时，坚决回退自然月口径，避免输出看似精确但并不可信的校正结果。

### 难点六：审批详情快捷跳转要快，但不能破坏当前操作上下文

问题：
如果直接把运营人员跳到另一个审批系统页签，虽然算“能看详情”，但会打断当前记录浏览上下文；如果只是给审批单号，又仍然需要用户自己检索。

解决：

- 在记录列表中保留审批单号联动。
- 采用 `Drawer` 而不是整页跳转，让运营人员继续留在申请记录上下文里。
- 同时对未同步记录做禁用态，避免制造新的“点击无响应”问题。

## 5. 为什么选择这版视觉表达

本次最终没有做大幅视觉重设计，而是刻意收敛到 Ant Design 5.x 官方默认风格，原因是：

1. 题目明确要求基于 Ant Design 5.x 组件规范。
2. 这是一个 B 端中后台效率型流程，可信度比“炫技感”更重要。
3. 题目明确要求基于 Ant Design 5.x 组件规范，严格执行官方默认组件表达更有说服力。
4. 笔试更看重交互严谨性、状态完整性和需求理解能力。

所以我选择在现有风格上做“秩序和层级”的升级，而不是换一套完全不同的视觉语言。
