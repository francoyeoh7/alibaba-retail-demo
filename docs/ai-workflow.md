# AI 工作流记录

## 1. 使用了哪些 AI 工具

- `Codex / GPT-5`：负责需求拆解、交互方案推演、React + Ant Design Demo 实现、测试编写与文档整理。

本次没有依赖额外的设计生成器或图像工具，主要原因是题目更强调业务理解、组件规范和交互完备度，可运行 Demo 的说服力更强。

## 2. Prompt 策略

我把任务拆成了 6 段 Prompt 目标：

1. `先理解 PRD`
   - 提取触发条件、校验逻辑、带出字段、成功后状态变化。
2. `再确定交付形式`
   - 不是只做静态稿，而是直接做一个可运行的 React + Ant Design Demo。
3. `最后补文档`
   - 把设计决策、组件理由、异常状态、AI 参与过程写成可交付文档。
4. `用测试约束核心流程`
   - 优先验证“按钮条件展示”“地址不一致阻断”“成功进入预填表单”三条主链路。
5. `根据参考图和追加反馈做结构补完`
   - 把 `点位信息待完善 -> 弹窗 -> 详情` 补成完整链路，再把 `员工维护/法人账号保护` 纳入同一平台导航。
6. `补齐剩余两条业务需求`
   - 增加 `校园经营分析` 和 `审批管理`，把订单统计口径校正与审批详情快捷跳转也落成可演示模块。

## 3. AI 迭代过程

### 第 1 轮：需求收敛

先把 PRD 转成结构化交互流：

- 列表页入口条件
- 转换确认弹窗
- 地址校验分支
- 老店升级表单字段来源
- 提交成功后的记录状态回写

这一轮的重点是避免漏掉“原单关闭”“点位释放”“跳过用户量/安全距离校验”这些容易被忽略的业务点。

### 第 2 轮：界面承载方案选择

在实现前比较了 3 种结构：

- 列表页 + 弹窗 + 独立表单页
- 列表页 + 抽屉补表单
- 列表页整合批量操作

最终选择第一种，因为最贴近题目截图，也最容易在笔试中清晰展示主流程和异常流。

### 第 2.5 轮：风格纠偏

在第一版 Demo 完成后，重新对照题目要求，发现虽然已经使用了 Ant Design 组件，但页面整体仍偏“自定义企业后台演绎”，距离 `严格按照 Ant Design 5.x` 还有差距。

因此又做了一轮风格纠偏：

- 去掉 Hero 区、渐变背景、大圆角、玻璃感和强化阴影
- 改回标准 `Layout + Card + Form + Table + Modal + Result`
- 查询区改成真正带 `Form.Item label` 的官方列表页结构
- 表单页改成标准分卡片表单，而不是作品化包装

### 第 3 轮：可运行 Demo 实现

基于 `Vite + React + TypeScript + Ant Design 5` 搭建 Demo，并加入了：

- 条件化操作按钮
- 风险确认弹窗
- 地址一致性校验
- 自动带出表单
- 提交后列表数据回写
- 演示场景切换

### 第 4 轮：根据反馈补全第二张参考图和员工维护模块

用户追加指出了两个关键缺口：

1. 参考图其实有两张，缺少 `点位信息待完善 -> 中间弹窗 -> 详情页` 这一步。
2. 平台还需要补一个 `员工维护` 模块，专门解决法人账号被误删/误停用的问题。

因此我新增了：

- 左侧导航中的 `员工维护`
- 状态入口弹窗
- `新店申请详情` 页面
- 法人账号保护弹窗
- 法人账号移交流程

### 第 5 轮：补齐校园统计和审批管理模块

用户继续补充剩余两个需求后，我又新增了两个模块：

- `校园经营分析`
  - 按日历月看
  - 剔除假期影响后看
  - 校历缺失回退逻辑
- `审批管理`
  - 申请记录与审批单号联动
  - 详情快捷抽屉
  - 审批详情待同步禁用态

这一轮的重点，不是简单“多加页面”，而是保证四个需求都能在同一套平台信息架构下成立。

### 第 6 轮：测试驱动修正

为了避免只做“静态可点击”的 Demo，我增加了 10 条自动化测试，覆盖：

- 满足条件时才显示 `转老店升级`
- 地址不一致时阻断并提示
- 地址一致时成功进入预填表单
- 点击 `点位信息待完善` 进入弹窗并跳到详情页
- 员工维护中显示当前法人账号
- 法人账号必须先移交，再允许后续停用
- 校园统计从 `按日历月看` 切到 `剔除假期影响后看`
- 校历缺失时回退自然月统计
- 运营后台从点位申请记录直接打开审批详情
- 审批详情待同步时显示禁用态

在测试过程中，AI 输出过几处需要人工修正的实现：

- 同时使用 toast 和行内错误提示，导致错误文案重复展示。
- 表单中同一地址既出现在点位地址输入框，也出现在详细地址输入框，造成信息冗余。
- Ant Design 的 Table / Modal 在测试环境中需要补少量兼容处理。
- 第一版视觉层级过强，不够像 Ant Design 官方默认风格，因此整体回收样式表达。
- 员工维护页顶部摘要最初是完整句子，自动化测试和演示讲解都更需要一个可单独识别的 `当前法人账号：XX` 摘要，所以又做了拆分。
- 校园统计页最初同时在统计卡片和表头里使用了 `自然月订单量`，自动化测试会命中多个节点，所以我把表头改成了更具体的 `原始订单量`。
- 审批管理页最初多个按钮都叫 `查看审批详情`，测试和真实使用都不利于区分，因此只保留主记录使用该文案，其余记录改成更具体的动作命名。

## 4. 我做了哪些人工修正，为什么

### 修正 1：保留企业后台调性，不做重视觉改造

原因：
题目给了现有业务截图，说明评审更关注“是否懂业务系统”，而不是“是否能做一版炫的作品页”。

### 修正 2：增加“演示场景”面板

原因：
PRD 要求覆盖正常流和异常/边界状态。如果每个状态都单独做页面，讲解成本很高；如果不做，又容易被认为覆盖不完整。

### 修正 3：把地址不一致校验放在弹窗确认阶段

原因：
这是用户成本最低的阻断点，能最大程度避免错误门店进入表单。

### 修正 4：把同一地址的重复展示收敛成“输入框 + 说明”而不是“双输入框”

原因：
重复字段会让页面显得像原型拼贴，不像真实业务表单。

### 修正 5：把法人账号保护做成“禁用危险操作 + 引导移交”，而不是只给一句提示

原因：
如果只是顶部写一条文案，并不能真正解决误操作问题。只有把危险按钮直接拿掉或禁用，再提供明确的移交路径，才算真正完成需求闭环。

### 修正 6：把校园数据问题做成“口径切换 + 校历管理”，而不是单纯加一张图表

原因：
这个需求的本质不是图表不够好看，而是统计分母出了问题。主页面应该只保留经营结果；校历来源、连续假期区间、官网导入结果和人工调整入口应该收进 `校历管理`。这样既避免主页面信息过载，也能让商户或运营在需要时追溯“系统为什么这样算”。

### 修正 7：审批详情使用抽屉承载，而不是新开整页

原因：
这条需求强调的是“快速跳转”，不是“重新导航”。使用 `Drawer` 可以在保留记录列表上下文的同时，把审批详情快速展开，更符合运营人员高频处理任务的节奏。

## 5. 如果再做一次，我会怎么优化 AI 工作流

1. 补一轮 `Playwright` 级的场景截图自动导出，把 Demo 页面直接转成可贴进汇报文档的视觉稿。
2. 增加一页“审批后状态对比板”，把转换前后记录变化做成对照视图，帮助评审更快理解业务闭环。
3. 提前把“平台级左导航 + 列表页 + 详情页 + 表单页 + 员工维护页 + 数据页 + 运营页”的壳子一开始就搭好，减少后续因范围扩大而重构页面结构。
